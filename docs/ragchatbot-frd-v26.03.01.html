In Scope

  • Document upload — PDF, DOCX, TXT
  • Text chunking and embedding (OpenAI)
  • Vector storage and retrieval (ChromaDB)
  • Conversational Q&A with chat history
  • Source attribution per answer
  • Multi-document support per session
  • Session clear (docs + chat)
  • React UI — upload panel + chat interface
  • Docker + docker-compose deployment
  • REST API with OpenAPI docs

Out of Scope (Phase 1)

  • User authentication / multi-user sessions
  • Cloud deployment (AWS / GCP / Azure)
  • Streaming responses
  • Fine-tuning or custom model training
  • Persistent chat history across restarts
  • OCR for image-only / scanned PDFs
  • Document management (list / delete individual docs)
  • Mobile / native app
  • Real-time collaborative sessions

System Description

The RAG Chatbot is a locally-deployed web application that allows a user to upload one or more documents, then ask natural language questions about their content. The system uses retrieval-augmented generation (RAG) to ground answers in the uploaded documents, with source attribution per answer and full conversational context across follow-up questions.

Phase 1 only: Single user, no auth, no cloud. All processing stays on the local machine. OpenAI API is the only external dependency.

F1.1 — File Upload & Validation

Req IDRequirementPriorityAcceptance Criteria
FR-1.1.1The system shall accept file uploads via POST /upload as multipart/form-dataMustEndpoint accepts .pdf, .docx, .txt and returns 200 with upload metadata
FR-1.1.2The system shall validate file type by extension before processingMust.exe, .zip, .jpg, unsupported extensions → 400 with allowed types listed
FR-1.1.3The system shall reject files exceeding 50MBMustFile >50MB → 413 before reading file bytes into memory
FR-1.1.4The system shall assign a unique doc_id (UUID4 short hex) to each uploadMustResponse includes doc_id; two uploads of same file produce different IDs
FR-1.1.5The system shall return upload metadata: {status, doc_id, filename, chunk_count, elapsed_ms}MustAll 5 fields present and non-null in every 200 response
FR-1.1.6The system shall handle duplicate uploads by replacing existing chunksMustRe-upload same file → old chunks deleted before new insert; chunk_count reflects new version

F1.2 — Document Parsing

Req IDRequirementPriorityAcceptance Criteria
FR-1.2.1The system shall extract text from PDF files using PyMuPDFMust5-page PDF → text extracted from all pages with page number metadata
FR-1.2.2The system shall extract text from DOCX files using python-docxMustDOCX with headings, paragraphs, tables → full text extracted
FR-1.2.3The system shall extract text from TXT files directly as UTF-8MustPlain text file → content read and chunked correctly
FR-1.2.4The system shall return 400 if no text can be extractedMustImage-only PDF → 400 "No text could be extracted from this document"
FR-1.2.5The system shall preserve page number metadata during PDF extractionMustEach chunk includes page field matching source page number (1-based)

F1.3 — Text Chunking

Req IDRequirementPriorityAcceptance Criteria
FR-1.3.1The system shall split text using LangChain RecursiveCharacterTextSplitterMustOutput is List[Document] with text content and metadata
FR-1.3.2Chunk size shall be 1000 charactersMustNo individual chunk exceeds 1000 characters
FR-1.3.3Chunk overlap shall be 200 charactersMustLast 200 chars of chunk N appear at start of chunk N+1
FR-1.3.4Each chunk shall carry metadata: doc_id, filename, page, chunk_index, upload_timestampMustAll 5 metadata fields populated on every ChromaDB entry

F1.4 — Embedding & Vector Storage

Req IDRequirementPriorityAcceptance Criteria
FR-1.4.1The system shall embed each chunk using OpenAI text-embedding-3-smallMustEach chunk → 1536-dim vector stored in ChromaDB
FR-1.4.2The system shall store embeddings in a ChromaDB collection named ragchatbot_docsMustCollection exists with correct entry count after upload
FR-1.4.3The system shall use deterministic chunk IDs: {doc_id}_{chunk_index}MustRe-uploading same content produces same IDs → upsert is idempotent
FR-1.4.4ChromaDB collection shall persist to disk at CHROMA_PERSIST_DIRMustServer restart → collection still queryable; no re-upload needed
FR-1.4.5The system shall log embedding elapsed time per uploadShouldelapsed_ms in response reflects actual time from receipt to ChromaDB confirm

F1.5 — Session Clear

Req IDRequirementPriorityAcceptance Criteria
FR-1.5.1The system shall delete all chunks from ChromaDB on POST /clearMustAfter /clear, ChromaDB collection is empty
FR-1.5.2The system shall recreate the empty collection immediately after deletionMustSubsequent /upload succeeds without server restart
FR-1.5.3The system shall return {status: "success", message: "All documents cleared"}Must200 response with both fields on every successful clear

F2.1 — Question Input & Validation

Req IDRequirementPriorityAcceptance Criteria

FR-2.1.2The system shall reject empty or whitespace-only questionsMustBlank question → 400 "Question cannot be empty"
FR-2.1.3The system shall return 400 if no documents are uploaded when a question is askedMustEmpty ChromaDB → 400 "No documents uploaded yet. Please upload a document first."

F2.2 — Chat History Management

FR-2.2.2The system shall pass chat history to LangChain in correct message formatMustLangChain receives correctly typed HumanMessage / AIMessage objects
FR-2.2.3Chat history shall be held client-side — not persisted on the serverMustServer restart resets to empty history; frontend maintains state in component

F2.3 — Question Condensation

Req IDRequirementPriorityAcceptance Criteria

FR-2.3.2Condensation shall use LangChain's ConversationalRetrievalChain condense stepMustChain logs show condensed question used for retrieval — not raw follow-up text

F2.4 — Vector Retrieval

Req IDRequirementPriorityAcceptance Criteria
FR-2.4.1The system shall retrieve the top 4 most semantically similar chunks per questionMustSources list has exactly 4 entries (or fewer if <4 chunks in collection)
FR-2.4.2Retrieval shall use ChromaDB cosine similarity searchMustReturned chunks are semantically relevant to the question
FR-2.4.3Retrieved chunks shall include full metadata: filename, page, chunk_index, chunk_textMustAll 4 metadata fields present on every source entry

F2.5 — LLM Answer Generation

FR-2.5.3The system shall return 502 if OpenAI API is unavailableMustOpenAI failure → 502 "LLM service temporarily unavailable. Try again."

UC-01 — Upload and Query a Single Document

Pre-conditionsApp running; no documents uploaded; ChromaDB collection empty
TriggerUser drags a PDF onto the upload zone
Main Flow1. User drags PDF → 2. POST /upload → 3. Parse, chunk, embed, store → 4. UI shows "contract.pdf — 42 chunks" → 5. User types question → 6. Retrieve top-4 chunks → 7. GPT-4o generates answer → 8. UI shows answer + sources
Alternate FlowFile rejected: step 2 returns 400 → UI shows inline error, no upload recorded
Exception FlowOpenAI unavailable at step 7 → UI shows "LLM service unavailable", input re-enabled
Related FRsFR-1.1.1–1.5.3, FR-2.1.1–2.6.3, FR-3.1.1–3.3.3

UC-02 — Ask a Follow-Up Question

Pre-conditionsAt least one document uploaded; at least one prior Q&A turn in chat thread
TriggerUser types "Can you give me more detail on that?"
Main Flow1. POST /ask with question + chat_history → 2. LangChain condenses to self-contained query → 3. Condensed query used for ChromaDB retrieval → 4. Answer contextually continues prior conversation
Exception FlowChat history malformed → 422 FastAPI validation error
Related FRsFR-2.2.1–2.3.2

UC-03 — Multi-Document Cross-Query

Pre-conditionsApp running with empty collection
TriggerUser uploads two documents, then asks a question spanning both
Main Flow1. Upload doc A → success → 2. Upload doc B → success → 3. Ask cross-document question → 4. Retrieval returns chunks from both docs → 5. Answer synthesises across both
Exception FlowOne upload fails → other doc still queryable; partial collection not corrupted
Post-conditionsSources panel shows chunks from both documents with correct filenames
Related FRsFR-1.1.6, FR-2.4.1–2.4.3, FR-2.6.1–2.6.3

UC-04 — Clear Session

Pre-conditionsOne or more documents uploaded; chat history present
TriggerUser clicks "Clear Session" button in header
Main Flow1. Confirmation dialog shown → 2. User confirms → 3. POST /clear → 4. ChromaDB deleted and recreated → 5. UI resets: empty chat, empty upload list, input disabled
Alternate FlowUser cancels confirmation → no action taken; state unchanged
Related FRsFR-1.5.1–1.5.3, FR-3.4.1–3.4.3
