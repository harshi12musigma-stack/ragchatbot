|---|---|

| Embeddings | OpenAI `text-embedding-3-small` |

#### Issue 2.1.2 — Define Component Interactions
**Type:** AGENT | **Owner:** Ved

- [ ] Define interaction: User → React UI → FastAPI `/upload` → LangChain loader → ChromaDB
- [ ] Define interaction: User → React UI → FastAPI `/ask` → LangChain retrieval chain → ChromaDB → OpenAI → answer
- [ ] Define interaction: User → React UI → FastAPI `/clear` → ChromaDB collection delete
- [ ] Define synchronous vs asynchronous calls (all sync in Phase 1)
- [ ] Define error propagation: how errors from OpenAI / ChromaDB surface to the UI
- [ ] Produce interaction diagram (sequence format) as structured markdown

#### Issue 2.1.3 — Select Architecture Pattern
**Type:** AGENT | **Owner:** Ved

- [ ] Confirm REST API pattern (not GraphQL, not gRPC) — rationale: simplicity for Phase 1
- [ ] Confirm local vector store pattern (not cloud-hosted Pinecone/Weaviate) — rationale: zero infra for 4-week build
- [ ] Confirm LangChain ConversationalRetrievalChain pattern — rationale: handles memory + retrieval in one abstraction
- [ ] Confirm stateless backend (session state held client-side as chat history array) — rationale: no DB needed
- [ ] Document pattern decisions with one-line rationale each

#### Issue 2.1.4 — Design API Contract
**Type:** AGENT | **Owner:** Ved

- [ ] Define `POST /upload` — request: `multipart/form-data` (file), response: `{status, doc_id, chunk_count, elapsed_ms}`
- [ ] Define `POST /ask` — request: `{question: str, chat_history: [{role, content}]}`, response: `{answer: str, sources: [{doc_id, chunk_text, page}]}`
- [ ] Define `POST /clear` — request: none, response: `{status, message}`
- [ ] Define `GET /health` — response: `{status: "ok"}`
- [ ] Define error response schema: `{error: str, detail: str, status_code: int}` for all 4xx/5xx
- [ ] Specify content-type headers, auth headers (none in Phase 1), CORS policy (allow localhost:3000)
- [ ] Write OpenAPI-compatible YAML or JSON stub for all 4 endpoints

#### Issue 2.1.5 — Define Data Flow
**Type:** AGENT | **Owner:** Ved

- [ ] Map upload flow step-by-step: file bytes → PyMuPDF/python-docx extract text → RecursiveCharacterTextSplitter → List[Document] → OpenAI embed each chunk → ChromaDB upsert with metadata {doc_id, filename, page, chunk_index}
- [ ] Map query flow step-by-step: question string + chat_history → LangChain ConversationalRetrievalChain → condense question (if follow-up) → ChromaDB similarity search (top_k=4) → retrieved chunks → GPT-4o prompt with context → answer string + source metadata
- [ ] Map clear flow: delete ChromaDB collection → recreate empty collection → confirm
- [ ] Identify data at rest: ChromaDB persist dir (local disk) — what's stored per chunk (embedding vector, text, metadata)
- [ ] Identify data in transit: file upload (multipart), OpenAI API calls (HTTPS)
- [ ] Document data flow as numbered step list (input to architecture diagram)

#### Issue 2.1.6 — Define ChromaDB Schema and Persistence Strategy
**Type:** AGENT | **Owner:** Ved

- [ ] Define collection name: `ragchatbot_docs`
- [ ] Define document metadata schema per chunk: `{doc_id: str, filename: str, page: int, chunk_index: int, upload_timestamp: str}`
- [ ] Define ChromaDB persist directory: `./chroma_db/` (configurable via `CHROMA_PERSIST_DIR` env var)
- [ ] Define collection lifecycle: created on first upload if not exists, persists across restarts, wiped only on `/clear`
- [ ] Define ID strategy per chunk: `{doc_id}_{chunk_index}` (deterministic, idempotent re-upload)
- [ ] Define what happens on duplicate upload: overwrite by doc_id prefix (delete old chunks, insert new)

#### Issue 6.1.5 — Document Known Limitations
**Type:** AGENT | **Owner:** Ved

- [ ] No authentication — single user only, do not expose publicly
- [ ] No streaming — full answer buffered before display
- [ ] Chat history resets on server restart (held in memory)
- [ ] OpenAI dependency — no offline mode
- [ ] Image-only PDFs (scanned, no text layer) not supported
- [ ] ChromaDB not suitable for >100k chunks without tuning
